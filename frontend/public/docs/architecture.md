# Technical Architecture

NEXUS Research Station utilizes a decoupled, registry-driven workspace architecture to maximize developer agility and keep the runtime lightweight.

```
┌──────────────────────────────────────────────┐
│                  Web UI                      │
│   (React / TypeScript / nexus-shell SPA)     │
└──────────────────────┬───────────────────────┘
                       │ HTTP/JSON
                       ▼
┌──────────────────────────────────────────────┐
│               Go Web Server                  │
│       (Embedded FS, CORS, API Routes)        │
└──────────────────────┬───────────────────────┘
                       │ Direct Compilation
                       ▼
┌──────────────────────────────────────────────┐
│           Single Binary Executable           │
│             (bin/nexus-research)             │
└──────────────────────────────────────────────┘
```

## 1. Registry-Driven Frontend
Instead of hardcoding layout layouts or page flows, the UI utilizes registries:
- **`componentRegistry`**: Allows custom components to be registered dynamically at boot time under a unique key.
- **`commandRegistry`**: Maps keybindings and executable commands.
- **`menuRegistry`**: Builds application menu trees dynamically from active components.
- **`useLayoutStore`**: Manages panel sizes, splits, active selections, and state serialization.

## 2. Compile-Time Go Embedding
The static assets generated by Vite are bundled inside the Go binary at compile-time:
```go
//go:embed all:frontend/dist
var frontendFS embed.FS
```
This guarantees that the application can be distributed as a single executable without requiring external asset directories.
